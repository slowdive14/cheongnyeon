# 청년정책 진단 플랫폼 — 문제해결 과정 (소프트웨어경진대회 발표문)

## 1. 무엇을 만들었나

청년이 자기 상황을 일상 언어로 적으면("월세가 부담돼요") 전국 2,633개 청년정책 가운데 맞는 정책을 의미 기반으로 찾아주고, 나이·지역·소득·모집기간 네 축으로 신청 자격까지 추정해주는 웹 서비스입니다. 설계할 때 세 가지 원칙을 세웠습니다. 먼저 위기 신호, 그러니까 자해나 자살 표현이 감지되면 검색보다 전문기관 안내를 먼저 보여줍니다. 자격 판정은 최대한 보수적으로 해서, 확인할 수 없는 조건은 "탈락"이 아니라 "확인 필요"로 분류합니다. 헛된 기대도, 부당한 배제도 막으려는 것입니다. 마지막으로 LLM이 만든 문장은 원문과 대조하는 검증, 곧 그라운딩을 통과해야만 사용자에게 보여줍니다.

구조는 React SPA(클라이언트)에 Supabase pgvector(벡터 DB), Edge Function(검색 서버), 그리고 적재 시점에 미리 계산해두는 Gemini를 얹었고, 사용자에게 API 키를 요구하지 않습니다. 검색만 서버가 맡고, 위기 감지와 자격 판정은 네트워크가 끊겨도 돌아가도록 클라이언트 순수 함수로 두었습니다.

오늘은 이 시스템을 완성하면서 만난 세 가지 문제를 어떻게 진단하고 해결했는지 말씀드리겠습니다.

## 2. 사례 1 — "월세"를 검색해도 아무것도 안 나온다

### 증상과 가설 소거

"월세 지원 받을 수 있는지 궁금해"라는, 가장 기본적인 질의에 결과가 0건이었습니다. "일자리", "개발자" 같은 정확한 키워드만 동작하는 듯 보였습니다. 의미검색이 이 서비스의 핵심 가치인 만큼 치명적인 결함이었습니다.

가설을 하나씩 실측으로 지워나갔습니다. 첫 번째, 질의 임베딩이 실패했나? 질의마다 결과가 달라지니 아니었습니다. 두 번째, 문서 임베딩이 붕괴했나? 저장된 벡터들의 코사인 유사도를 직접 계산해보니 0.58~0.69로 건강하게 분산돼 있어 이것도 아니었습니다. 세 번째, 질의와 문서의 정렬이 어긋났나? "월세" 질의 벡터는 실제로 주거 정책(0.70)에 일자리 정책(0.60)보다 가까웠습니다. 역시 아니었습니다.

그런데 몇 분 뒤 같은 요청이 정상 결과를 돌려줬습니다. 같은 입력에 다른 출력, 진짜 증상은 비결정성이었던 겁니다.

### 근본 원인 두 겹과 해결

원인은 두 겹이었습니다. 하나는 pgvector의 HNSW 근사 최근접 탐색인데, 기본 탐색 폭(ef_search=40)이 낮다 보니 DB가 한동안 쉬다 깨어난 콜드 상태에서는 그래프 진입점 근처의 "허브" 문서만 돌려줬습니다. 어떤 질의든 같은 클러스터로 쏠리던 현상이 여기서 나온 것입니다. 탐색 폭을 120으로 올려 재현율을 안정화했습니다.

다른 하나는 구조 자체에 숨어 있었습니다. 기존 설계는 벡터로 후보 50건을 뽑은 다음 그 안에서 키워드로 재정렬하는 방식이라, 벡터가 관련 문서를 후보에 넣지 못하면 키워드가 구제할 길이 아예 없었습니다. 그래서 키워드 검색을 독립된 후보 팔로 떼어내 벡터 후보와 합집합(UNION)으로 묶는 하이브리드 구조로 다시 설계했습니다.

### 수정이 만든 2차 회귀와 재조정

고치고 나니 이번엔 "우울하고 무기력해요 상담" 질의의 1위가 "전세사기 피해예방 교육"으로 나오는 회귀가 생겼습니다. 키워드 가중치 0.4가 너무 커서, "상담"이라는 흔한 토큰 하나가 의미 유사도가 훨씬 높은 문서(고민상담소 0.69 vs 전세사기교육 0.56)를 뒤엎어버린 겁니다. 그래서 순위는 코사인 유사도가 지배하게 두고, 키워드는 최대 +0.05의 타이브레이커로만 작동하도록 낮췄습니다. 배포하기 전에 대표 질의 6개로 새 랭킹 공식을 시뮬레이션해 모든 질의가 나아졌음을 확인한 뒤 반영했고, 마지막으로 12개 질의 배터리와 같은 질의를 반복 실행하는 결정성 검증까지 거쳤습니다.

## 3. 사례 2 — 지역을 입력하자 결과가 사라졌다

시·도와 나이를 입력받아 자격 판정을 정밀화하는 기능을 TDD로 구현했습니다. 테스트 669개를 통과하고, 코드 리뷰·안전 감사·통합 QA 3중 검수도 모두 통과했습니다. 그런데 실제 브라우저에서 "부산, 25세"를 설정하고 월세를 검색하자 결과가 0건이 됐습니다. 지역을 넣기 전에는 나오던 결과가, 넣으니 사라진 것입니다.

네트워크 응답을 열어보니 원인이 드러났습니다. 검색이 돌려준 상위 10건이 전부 울산·제주·경북의 월세 정책이었고, 클라이언트 자격 엔진이 이를 전부 "지역 불일치 → 차단"으로 걸러낸 것입니다. DB에 "부산 청년 월세 지원"이 엄연히 있는데도, 검색이 사용자 지역을 모르니 상위 10자리를 타 지역 정책이 차지해버린 셈입니다.

해결의 열쇠는 계층별 책임을 다시 나누는 데 있었습니다. 검색(후보 선정)과 자격 판정(권위 판정)은 그대로 분리하되, 후보 선정 단계에 지역 양립성 필터를 넣었습니다. 사용자 지역과 양립할 수 없는 정책이 후보 정원을 낭비하지 않게 한 것입니다. 이때도 보수 원칙은 지켜서, 전국 정책과 지역 미상 정책은 배제하지 않았습니다. 미상은 "확인 필요"로 노출돼야 하니까요. 자격 판정 권위는 여전히 클라이언트 엔진에 있으니 이중으로 방어되는 셈입니다.

이 사례에서 교훈을 하나 얻었습니다. 테스트와 검수는 "명세대로 구현됐는가"는 검증하지만, "명세 자체가 옳은가"는 실사용 검증에서야 드러난다는 것입니다. 3중 검수가 놓친 결함을 라이브 5분이 잡았고, 이후 라이브 e2e를 필수 게이트로 승격했습니다.

## 4. 사례 3 — AI 설명인 줄 알았던 고정 템플릿

결과 카드의 "왜 맞을까요" 설명이 모든 정책에서 똑같은 틀("OO 정책과 관련이 있어 보여요…")로 나오는 문제를 추적했습니다. 알고 보니 이 문구는 LLM이 만든 게 아니라 장애 폴백이었습니다. 대량 적재 당시 API 지출 한도 초과(429)로 설명 생성이 실패하면서 안전 폴백 문구가 저장됐는데, 이후 보충 로직이 "null인 것만 재생성"하도록 돼 있어 폴백 문구가 유효한 값으로 취급됐고, 결국 2,633건 전체에 그대로 굳어버린 것입니다. 장애의 2차 효과가 데이터에 각인된 사례였습니다.

더 근본적인 한계도 확인했습니다. 비용 때문에 설명을 적재 시점에 미리 계산하는데, 그 시점에는 사용자의 질문이 아직 없으니 "왜 당신의 질문에 맞는지"는 원리적으로 쓸 수가 없습니다. 그래서 대안을 다시 설계했습니다. 하나는 자격 엔진의 판정 근거를 체크리스트로 그대로 노출하는 것입니다("나이 19~34세 — 내 나이 충족 / 소득 조건 — 원문 확인"). 추론이 아니라 계산 결과라 환각이 원리적으로 불가능하고 비용도 0입니다. 다른 하나는 미리 계산하는 설명의 목적을 "질의 관련성"에서 "혜택 요약"으로 바꾼 것입니다("월세 매달 최대 20만원, 12개월 지원"). 질의에 기대지 않는 정보만 미리 계산하자는 것이죠. 미리 계산할지 실시간으로 계산할지의 경계를 "사용자 입력에 의존하느냐"로 정한 것, 이것이 이번 재설계의 핵심입니다.

## 5. 프로세스와 성과

구현은 계획 → TDD 구현 → 코드 리뷰·안전 감사·통합 QA 병렬 검수 → 라이브 검증의 파이프라인으로 진행했고, 자격·위기·생성처럼 안전에 직결되는 모듈은 감사를 생략하지 않는 것을 원칙으로 삼았습니다. 지금 테스트 669개가 전부 통과하고, 타입·린트·빌드가 무결하며, 도메인 커버리지는 93%입니다. 전량 적재(2,633건, 벡터 1536차원에 설명까지) 실측 비용은 약 $1이었고, 증분 처리 덕분에 유지 비용은 0에 수렴합니다.

남은 과제는 네 가지입니다. 데이터를 자동으로 신선하게 유지할 정기 배치, 서울시 자체 정책 약 1,000건 수집(지금은 서울 한정 60건뿐이라 그 공백을 메워야 합니다), 공개 전 보안 마감(CORS 제한·레이트리밋), 그리고 무기력한 사용자를 신청 완료까지 데려가는 UX("오늘은 이것만" 하고 과제를 쪼개거나, 지역 청년센터로 연결하는 방식)입니다. 감사합니다.
