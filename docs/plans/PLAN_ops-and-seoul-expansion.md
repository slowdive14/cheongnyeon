# PLAN: 운영 자동화 + 서울 소스 확장 + 공개 배포 마감

작성: 2026-07-01 · 상태: 계획(사용자 승인 대기) · 선행: PLAN_scope-expansion.md (스코프 확장 e2e 완료, 프로필 입력 완료)

## 목표
1. **데이터가 스스로 신선해지는 플랫폼**: 온통청년 API(갱신주기 "실시간", 공공데이터포털 15143273)를 정기 배치로 자동 재수집.
2. **서울 커버리지 확장**: 청년몽땅정보통(youth.seoul.go.kr)의 서울시 326건+자치구 675건(≈1,000건)을 수집해 현재 서울 한정 60건의 공백을 메움.
3. **공개 배포 마감**: R2(편의)·D1(코드 보험)·C4(공개 배포 전 자물쇠)로 안전하게 오픈.

## 현재 상태 (전제)
- Supabase 2,633건(벡터 1536d·설명 precompute) + 검색 Edge Function 라이브. 자격·위기는 클라.
- 인제스트는 **수동**(`npm run ingest`, 로컬 PC) — 증분(contentHash) 지원으로 변경분만 LLM 비용 발생.
- Git 저장소 아님. Vercel 미배포. Edge Function CORS `*`.
- 다중 출처 대비 설계 완료: `Policy.source`, 1차 키 source+id, 2차 유사도(≥0.85) 수동 후보.

## Phase A — 배포 기반: GitHub + 자동 배치 (선행 필수)

**A1. Git/GitHub 준비**
- `git init` + `.gitignore` 정밀 점검(`.env`·키·대용량 캐시 제외 — **비밀이 커밋 이력에 절대 안 들어가게 최우선**).
- GitHub 비공개 저장소 생성·push. (비공개 권장 — 코드에 비밀은 없지만 공격면 축소.)

**A2. GitHub Secrets 등록** (운영자)
- `ONTONG_API_KEY`, `GEMINI_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`.

**A3. 배치 워크플로 (GitHub Actions cron)**
- 주기 기본 **주 2회**(월·목 새벽) — 결정 D-1.
- 실행: checkout → npm ci → `npm run ingest`(증분, regionScope=all).
- 산출: 실행 요약(총/신규/변경/재파싱 건수) 로그. **실패 시 알림**(Actions 실패 메일 기본 + 요약 코멘트).
- 가드: 타임아웃·동시 실행 방지(concurrency 그룹)·Gemini 429 시 재시도는 기존 코드 재사용.

**A4. Vercel 연결(프리뷰)**
- GitHub 연동 → 빌드 env(`VITE_SEARCH_FN_URL`, `VITE_SUPABASE_ANON_KEY`).
- **공식 공개(URL 공유·홍보)는 Phase D의 C4 완료 후.** 그 전까지는 개인 확인용.

**설계 결정 필요**
- **D-1 배치 주기**: 주 2회(권장, 비용↓) vs 매일 새벽(신선도↑, 변경분 적어 비용 차이 크지 않음).
- **D-2 스테일 정책 처리**: 온통청년에서 내려간(더는 안 오는) 정책 처리 방침.
  - (가) 유지 + 카드 '최종 업데이트' 노출(현행) — 위험: 종료 정책이 계속 검색됨.
  - (나) 마지막 fetchedAt이 N회 연속 갱신에서 빠지면 검색 제외(soft delete) — **권장(보수 원칙: 확인 불가한 정보는 노출 강도 낮춤)**.

**게이트**: 배치 1회 실운영 성공(Actions 로그로 증분 동작 확인), 비밀 커밋 이력 0(스캔).

## Phase B — 서울 청년몽땅 수집기 (`source='seoul-youth'`)

**B0. 정찰 (착수 전 반드시)**
- 목록/상세 페이지 구조 분석 — 내부 JSON 엔드포인트 존재 확인(있으면 스크래핑 대신 그걸 사용, 훨씬 견고).
- 상세 페이지에서 자격 필드(연령·소득·기간·지역/자치구) 추출 가능성 확인.
- 공공누리 이용조건·robots 확인. **자치구 정책 포함 범위 확정(결정 D-3, 포함 권장)**.
- 산출: 정찰 보고(`_workspace/`) — 여기서 실현 불가 판정이 나면 Phase B 중단하고 대안(정보공개청구로 API 요청 등) 보고.

**B1. SeoulClient + 정규화기**
- `IngestClient` 구현(fetchAll: 서울시+자치구 섹션만 — **'중앙정부/타지역' 섹션은 온통청년 데이터 자체(같은 plcyBizId)이므로 원천 제외 = 중복 0**).
- `normalizeSeoulPolicy`: 서울 스키마 → `Policy`(regionCodes=['11'], 자치구는 regionText 보존). 원문 링크는 청년몽땅 상세 URL.

**B2. 파이프라인 편입**
- 기존 ingest에 다중 클라이언트 합류(fetchAll 병합) — 증분·설명·임베딩 precompute 동일 재사용.
- 2차 유사도 후보(온통청년 서울분 60건과의 겹침) 리포트 확인 → 수동 검토.
- 카드에 출처 표기(온통청년/청년몽땅) — 신뢰 표면(원문 확인 유도).

**B3. 배치 합류 + 라이브 검증**
- Phase A 배치에 서울 수집 포함. 브라우저: 서울 선택 + 대표 질의(월세·수당·자치구 정책)로 실측.

**안전 포인트(감사 축)**: 수집 실패가 기존 온통청년 데이터를 지우지 않게(부분 실패 격리), 자격 불명 필드는 전부 review 보수 판정, 원문 링크 정확성.

**게이트**: 서울시+자치구 ≈1,000건 적재, 중복 후보 리포트 검토 완료, 테스트·tsc·lint 그린, 라이브 검증.

## Phase C — 마감 3종: "R2 = 편의, D1 = 코드 보험, C4 = 공개 배포 전 자물쇠"

**C-D1. 훅 단위 테스트 (코드 보험, 최소 작업)**
- `useProfileState` 직접 실행 테스트(초기값·patch 병합·income 보존·undefined 명시 병합). QA 지적(38_qa D1) 해소.

**C-R2. 프로필 localStorage (편의)**
- 시·도·나이 새로고침 유지. `useProfileState` 훅 경계에 이미 삽입 지점 마련됨(App 변경 0).
- 안전: 저장 값도 로드 시 파서 재검증(오염 값 유입 차단).

**C-C4. 공개 배포 전 자물쇠 (오픈 게이트)**
- (a) 마운트 시 불필요 원격 검색 1회 제거(실질의 있을 때만).
- (b) Edge Function CORS `*` → Vercel 도메인 제한.
- (c) 레이트리밋(IP 단위 단순 제한 — Gemini 비용 폭탄 방지).
- (d) near-dup 결과 표시 정리(수동 후보 ~400 검토 반영).
- (e) 안전 재감사(위기 layer-1·'추정' 고지·원문 링크 — 서울 소스 포함 전영역).
- **완료 후에만 공식 공개.**

**게이트**: safety-domain-auditor 재감사 승인 + 전체 게이트 그린 → 공개.

## Phase D — 결과 카드 설명 개선 ("왜 맞을까요" 교체)

**배경(진단, 2026-07-01)**: 현재 카드의 "왜 맞을까요 … (추정)" 문구는 LLM 산출이 아니라 **폴백 템플릿**이다. 전량 적재 당시 Gemini 지출한도 429로 설명 생성이 실패 → `explainMatch`의 안전 폴백이 저장됐고, 백필은 "explanation이 null인 것만" 보강하는 로직이라 폴백 문구가 '채워진 값'으로 취급돼 재생성이 막혔다(ingest.ts 6a). 또한 구조적 한계: precompute 설명은 질의 무관이라 "왜 당신 질문에 맞는지"는 원리적으로 쓸 수 없다.

**D-①. 자격 체크리스트 UI ("나와 맞는 점") — LLM 불필요, 독립 작업(즉시 가능)**
- 자격 엔진의 판정 근거(reasons + 통과 축)를 카드에 체크리스트로 노출:
  - ✓ 통과 축(예: "나이 19~34세 — 내 나이 25세 충족", "부산 거주자 대상 — 내 지역 일치")
  - ? review 축(예: "소득 조건 — 원문에서 확인")
- 프로필(시·도·나이) 입력과 자동 연동 = 진짜 개인화. 계산 결과라 환각 0·비용 0.
- **기계적 "왜 맞을까요" prose 제거.** '추정' 고지·원문 링크는 유지(안전 표면 불변).
- 구현 메모: evaluateOne이 현재 blocked/review reasons만 반환 — 통과 축 표시용으로 pass 축 정보 노출 필요(EvaluatedPolicy 확장 or 카드에서 정책 필드+프로필로 재구성). 안전 감사 축: 체크 표기가 자격 단정("자격이 됩니다")으로 읽히지 않게 문구 보수성 유지.

**D-②. 혜택 한 줄 precompute — 목적 변경 재생성(1회 ~$1), Phase A 배치와 묶음**
- explain 프롬프트 목적 변경: "왜 관련 있는지"(질의 없인 공허) → **"무엇을 주는 정책인지 쉬운 말 1문장"**(예: "월세를 매달 최대 20만원, 12개월까지 지원").
- 기존 그라운딩 검증(원문에 없는 숫자·지역·단정 거부) 그대로 적용.
- **백필 버그 수정 동반**: 폴백 문구는 저장하지 않기(null 저장 → 다음 run 재시도) 또는 source=fallback 표시 저장. 기존 2,633건 폴백 문구는 재생성 대상.
- 카드의 요약 슬롯을 혜택 한 줄로 교체(관공서 요약문 대체).

**D-③. 질의 맞춤 설명 on-demand — 백로그(C-C4 레이트리밋 이후)**
- 카드에서 사용자가 탭할 때만 Edge Function이 질의+정책으로 실시간 생성(검색마다 아님 → 비용 통제).

**게이트**: D-①은 safety-domain-auditor 감사(자격 단정 표현 금지·고지 불변) 필수. D-②는 재생성 샘플 그라운딩 검증 통과 확인.

## Phase E — UI 리디자인 (청년 친화 톤) — Phase D와 묶어 진행 권장

**★디자인 SSOT: 루트 `DESIGN.md`** (2026-07-02 작성). 색 토큰(크림/클레이/상태 3종)·모양·컴포넌트 인벤토리·확정 카피 표·접근성 바닥선(KRDS 참조)·안전 불변 조항이 전부 코드화돼 있다. Phase E·F의 모든 UI 구현은 이 문서를 근거로 하고, 아래 E-①~③은 요약이다(충돌 시 DESIGN.md 우선). Figma는 "보여주는 곳"(발표 자산·시안 탐색)으로만 사용.

**방향(사용자 시안 확인, 2026-07-01)**: "발랄함"이 아니라 **"다정함"** — 핵심 사용자(지치고 무기력한 상태 포함)에게 과한 명랑함은 부담. 관공서 서식 → 대화.

**E-①. 카피(말투) 전환 — 비용 0, 최대 레버**
- 헤더: "청년정책 진단" → "요즘 어때요?"(인사 = 입력 유도) + "지금 상황을 편하게 적어주면, 맞는 정책을 찾아드려요".
- 입력 placeholder를 실제 예문으로("자취하는데 월세가 벅차요…"), 버튼 "정책 찾기" → "내 정책 찾기".
- 결과 헤드라인: "'…' 추정 결과" → "상황에 맞을 만한 N개를 찾았어요"(개수 = 성취감).
- 빈 결과: "지금 바로 맞는 정책을 찾지 못했어요" → "이 방향으론 못 찾았어요. 이런 쪽은 어때요?"(대안 유도 유지).
- 상태 라벨 인간화: "지금 신청 가능"→"지금 바로 신청돼요" / "거의 충족"→"거의 다 왔어요 — N만 확인" / "자격 확인 필요"→"몇 가지만 확인하면 돼요".

**E-②. 비주얼 시스템**
- 배경 회색(slate) → 따뜻한 크림톤, 포인트 하늘색(sky) → 클레이/코랄 계열. 모서리 더 둥글게(카드 16~20px).
- 예시 칩 → 꼬리 달린 말풍선 + 카테고리 라인 아이콘(하트·집·서류가방·지갑) + 섹션 라벨 "이렇게 적는 분들이 많아요"(예시→동질감).
- 프로필 입력(시·도+나이 서식 2칸) → "부산 · 25세 ✏️" 알약 1개(탭하면 펼침).

**E-③. 상태·신뢰 마감**
- 검색 로딩: "2,633개 정책을 뒤지는 중…"(신뢰+재미). 하단 신선도 한 줄: "어제 업데이트된 전국 N개 정책에서 찾아드려요"(Phase A 배치 후 실데이터).

**★안전 불변(엄수)**: 위기 배너·"추정" 고지·원문 링크·위기 푸터 문구는 톤 전환 대상에서 제외(신뢰 표면 — 진지함 유지). 문구 변경이 위기 라우팅·자격 표시 의미를 바꾸지 않는지 safety-domain-auditor 감사 필수. 상태 라벨 변경은 기존 테스트의 라벨 단언 전면 갱신 동반.

**게이트**: safety-domain-auditor 승인(안전 표면 불변·자격 단정 없음) + 전체 테스트 그린 + 브라우저 라이브 확인.

**F-③. 청년센터 연결(사람 연결) — 난이도 하, 정적 테이블**
- "혼자 하기 버거우면 OO청년센터가 같이 해줘요" — 사용자 시·도(프로필)에 맞는 청년(지원)센터 전화·카톡 채널 연결. 17개 시·도 정적 테이블(기존 SIDO 인프라에 꽂음). 위기 푸터와 별개 층위(위기=전문기관, 동행=청년센터). 상담자 사용자층에도 유용.

**F-④. 내 신청함 — 난이도 중, R2와 동일 인프라(localStorage)**
- 관심 정책 저장 + 진행 단계 체크. 재방문 시 "지난번에 보던 OO, 마감까지 N일" 리마인드(알림 인프라 없이 재방문 표시부터). 단계 체크 시 절제된 응원 카피 1줄.

**F-⑤. 신청 로드맵 + "오늘은 이것만" — 난이도 중, D-② 재생성 배치에 편승**
- 카드 펼침 시 신청 절차 3단계 체크리스트(① 서류 준비 ② 신청 방법 ③ 결과 대기) + **최소 첫 걸음 1개**("오늘은 이것만: 등본 1통, 정부24에서 5분") — 무기력 UX 핵심 = 과제 쪼개기.
- 재료: `parsed.chunks.application`(신청방법 청크)이 이미 파이프라인에 존재하나 현 데이터엔 대부분 비어있음 → **D-② 재생성 배치에서 함께 채우고 구조화**(단계 JSON precompute). 그라운딩 검증 동일 적용(원문에 없는 서류·기관·기간 날조 거부).

**★안전 불변(엄수)**: 로드맵·첫 걸음이 자격 단정("신청하면 됩니다")으로 읽히지 않게 — "추정 + 원문 확인" 고지 유지. 서류·마감 정보도 원문 우선 명시. 위기 상태에서는 신청 동행 UI도 미렌더(위기 배너 단독 불변식 유지).

**게이트**: safety-domain-auditor 감사(단정 금지·위기 불변식·날조 거부) + F-⑤ 재생성 샘플 그라운딩 검증 + 전체 테스트 그린 + 라이브 확인.
