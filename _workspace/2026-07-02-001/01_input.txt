# 청년정책 진단 플랫폼 — 문제해결 과정 (소프트웨어경진대회 발표문)

## 1. 무엇을 만들었나

청년이 자기 상황을 일상 언어로 적으면("월세가 부담돼요") 전국 2,633개 청년정책에서 맞는 정책을 의미 기반으로 검색하고, 나이·지역·소득·모집기간 4개 축으로 신청 자격까지 추정해주는 웹 서비스입니다. 핵심 설계 원칙은 세 가지입니다. 첫째, 위기 신호(자해·자살 표현)가 감지되면 검색보다 먼저 전문기관 안내로 라우팅합니다. 둘째, 자격 판정은 보수적으로 — 확인 불가능한 조건은 "탈락"이 아니라 "확인 필요"로 분류해 헛된 기대와 부당한 배제를 모두 막습니다. 셋째, LLM이 생성하는 모든 문장은 원문 대조 검증(그라운딩)을 통과해야 사용자에게 노출됩니다.

구조는 React SPA(클라이언트) + Supabase pgvector(벡터 DB) + Edge Function(검색 서버) + Gemini(적재 시 사전 계산)이며, 사용자에게 API 키를 요구하지 않습니다. 검색만 서버가 하고, 위기 감지와 자격 판정은 네트워크가 끊겨도 동작하도록 클라이언트 순수 함수로 두었습니다.

오늘은 이 시스템을 완성하는 과정에서 만난 세 가지 문제와, 각각을 어떻게 진단하고 해결했는지 말씀드리겠습니다.

## 2. 사례 1 — "월세"를 검색해도 아무것도 안 나온다

### 증상과 가설 소거

"월세 지원 받을 수 있는지 궁금해"라는 가장 기본적인 질의에 결과가 0건이었습니다. 정확한 키워드("일자리", "개발자")만 동작하는 것처럼 보였습니다. 의미검색이 서비스의 핵심 가치라서 치명적인 결함이었습니다.

가설을 하나씩 실측으로 소거했습니다. ① 질의 임베딩 실패? — 질의마다 결과가 달라지므로 기각. ② 문서 임베딩 붕괴? — 저장된 벡터들의 코사인 유사도를 직접 계산해 0.58~0.69로 건강하게 분산됨을 확인, 기각. ③ 질의-문서 정렬 붕괴? — "월세" 질의 벡터는 실제로 주거 정책(0.70)이 일자리 정책(0.60)보다 가까웠습니다. 기각.

그런데 몇 분 뒤 동일 요청이 정상 결과를 반환했습니다. 같은 입력, 다른 출력 — 비결정성이 진짜 증상이었습니다.

### 근본 원인 두 겹과 해결

첫째, pgvector의 HNSW 근사 최근접 탐색은 기본 탐색 폭(ef_search=40)이 낮아, DB가 유휴 후 깨어난 콜드 상태에서 그래프 진입점 근처의 "허브" 문서만 반환했습니다. 어떤 질의든 같은 클러스터로 쏠리는 현상이 여기서 나왔습니다. 탐색 폭을 120으로 올려 재현율을 안정화했습니다.

둘째, 구조적 결함이 숨어 있었습니다. 기존 설계는 "벡터로 후보 50건 → 그 안에서 키워드 재정렬"이라, 벡터가 관련 문서를 후보에 넣지 못하면 키워드가 구제할 경로 자체가 없었습니다. 키워드 검색을 독립 후보 팔로 분리해 벡터 후보와 합집합(UNION)하는 하이브리드 구조로 재설계했습니다.

### 수정이 만든 2차 회귀와 재조정

수정 직후 "우울하고 무기력해요 상담" 질의의 1위가 "전세사기 피해예방 교육"이 되는 회귀가 나타났습니다. 키워드 가중치 0.4가 과해서, "상담"이라는 흔한 토큰 하나가 의미 유사도가 훨씬 높은 문서(고민상담소 0.69 vs 전세사기교육 0.56)를 뒤엎은 것입니다. 순위는 코사인 유사도가 지배하고 키워드는 최대 +0.05의 타이브레이커로만 작동하도록 낮췄습니다. 배포 전 6개 대표 질의로 새 랭킹 공식을 시뮬레이션해 전 질의 개선을 확인한 뒤 반영했고, 최종적으로 12개 질의 배터리와 동일 질의 반복 실행(결정성)으로 검증했습니다.

## 3. 사례 2 — 지역을 입력하자 결과가 사라졌다

시·도와 나이를 입력받아 자격 판정을 정밀화하는 기능을 TDD로 구현했습니다. 테스트 669개 통과, 코드 리뷰·안전 감사·통합 QA 3중 검수를 모두 통과했습니다. 그런데 실제 브라우저에서 "부산, 25세"를 설정하고 월세를 검색하자 결과가 0건이 됐습니다. 지역을 입력하기 전에는 나오던 결과가, 입력하니 사라진 것입니다.

네트워크 응답을 열어보니 원인이 보였습니다. 검색이 반환한 상위 10건이 전부 울산·제주·경북의 월세 정책이었고, 클라이언트 자격 엔진이 전부 "지역 불일치 → 차단"으로 걸러냈습니다. DB에 "부산 청년 월세 지원"이 존재하는데도, 검색이 사용자 지역을 모르니 상위 10개 자리를 타 지역 정책이 잠식한 것입니다.

해결은 계층 책임의 재정의였습니다. 검색(후보 선정)과 자격 판정(권위 판정)은 분리하되, 후보 선정 단계에 지역 양립성 필터를 추가했습니다 — 사용자 지역과 양립 불가한 정책이 후보 정원을 낭비하지 않도록. 이때 보수 원칙을 유지해 전국 정책과 지역 미상 정책은 배제하지 않았습니다(미상은 "확인 필요"로 노출되어야 하므로). 자격 판정 권위는 여전히 클라이언트 엔진에 있어 이중 방어가 됩니다.

이 사례에서 얻은 교훈이 하나 있습니다. 테스트와 검수는 "명세대로 구현됐는가"를 검증하지만, "명세 자체가 옳은가"는 실사용 검증에서만 드러납니다. 3중 검수가 통과시킨 결함을 라이브 5분이 잡았고, 이후 라이브 e2e를 필수 게이트로 승격했습니다.

## 4. 사례 3 — AI 설명인 줄 알았던 고정 템플릿

결과 카드의 "왜 맞을까요" 설명이 모든 정책에서 같은 틀("OO 정책과 관련이 있어 보여요…")로 나오는 문제를 추적했습니다. 확인 결과 이 문구는 LLM 산출이 아니라 장애 폴백이었습니다. 대량 적재 당시 API 지출 한도 초과(429)로 설명 생성이 실패하며 안전 폴백 문구가 저장됐고, 이후 보충 로직이 "null인 것만 재생성"하는 조건이라 폴백 문구가 유효값으로 취급돼 2,633건 전체에 고착된 것입니다. 장애의 2차 효과가 데이터에 각인된 사례입니다.

더 근본적인 한계도 확인했습니다. 비용 문제로 설명을 적재 시 사전 계산하는데, 사전 계산 시점에는 사용자의 질문이 존재하지 않으므로 "왜 당신의 질문에 맞는지"는 원리적으로 쓸 수 없습니다. 그래서 대안을 재설계했습니다. ① 자격 엔진의 판정 근거를 체크리스트로 직접 노출("나이 19~34세 — 내 나이 충족 / 소득 조건 — 원문 확인") — 추론이 아닌 계산 결과라 환각이 원리적으로 불가능하고 비용이 0입니다. ② 사전 계산 설명은 목적을 "질의 관련성"에서 "혜택 요약"("월세 매달 최대 20만원, 12개월 지원")으로 변경 — 질의에 의존하지 않는 정보만 사전 계산합니다. 사전 계산과 실시간의 경계를 "사용자 입력 의존 여부"로 정의한 것이 이 재설계의 핵심입니다.

## 5. 프로세스와 성과

구현은 계획 → TDD 구현 → 코드 리뷰 + 안전 감사 + 통합 QA 병렬 검수 → 라이브 검증의 파이프라인으로 진행했고, 안전 직결 모듈(자격·위기·생성)은 감사를 생략하지 않는 것을 원칙으로 했습니다. 현재 테스트 669개 전체 통과, 타입·린트·빌드 무결, 도메인 커버리지 93%입니다. 전량 적재(2,633건, 벡터 1536차원 + 설명) 실측 비용은 약 $1로, 증분 처리 덕에 유지 비용은 0에 수렴합니다.

남은 과제는 정기 배치(데이터 자동 신선화), 서울시 자체 정책 약 1,000건 수집(현재 서울 한정 60건의 공백 해소), 공개 전 보안 마감(CORS 제한·레이트리밋), 그리고 무기력한 사용자를 신청 완료까지 동행시키는 UX("오늘은 이것만" 과제 쪼개기, 지역 청년센터 연결)입니다. 감사합니다.
